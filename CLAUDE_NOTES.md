# Project Notes

- `skills/writing-skills/render-graphs.js` currently fails under the root `"type": "module"` package setting because it uses CommonJS `require`. Graphviz `dot` is also not installed in this environment, so DOT diagram verification may need a script fix or a different local setup.
- For Git-installed Codex marketplaces where the plugin lives at the repository root, use a marketplace plugin source of `"source": "url"` with the repository URL and ref. A repo-cloned marketplace entry using `"source": "local", "path": "./"` can be registered by `codex plugin marketplace add` but then skipped in `/plugins`.
- Skills that resolve a trunk/base branch must **detect** it, never assume `main`. This user deliberately configures git to use `master` as the default branch for new repos. Detect with `git rev-parse --verify --quiet master` / `main` (use whichever exists; ask if both or neither), not by hardcoding `main` or trying it first. Applied in `external-code-review` (Step 2) and `finishing-a-development-branch` (Step 2) in v1.4.0.
- Skill-triggering tests (`tests/skill-triggering/run-test.sh`) shell out to a real `claude -p --plugin-dir <repo-root>`, which loads skills from the **working tree** — no plugin reload needed to test an in-progress skill. Each run can take minutes (300s timeout). The description frontmatter is what drives triggering, so editing a skill's *body* after a GREEN pass does not require re-running the trigger test.
- Skill prose should describe platform capabilities, not exact tool names, when the same concept exists across agent harnesses. Exact names such as `TodoWrite`, `update_plan`, `write_todos`, or `todowrite` belong in mapping/bootstrap docs as known examples; the core skill should tell the agent to discover the available runtime tool and fall back gracefully.
- Treat user-supplied reference docs for a task as session input, not repo documentation, unless the user explicitly asks to keep them. External CLI docs can stale quickly; implement against them, but do not commit local copies or README links to those copies by default.
- **Comparing against upstream Superpowers:** the fork point is upstream tag `v5.0.6` (commit `eafe962`, 2026-03-25) — it exists as a tag in the upstream repo, so `git diff v5.0.6..HEAD -- <path>` there separates upstream improvements from this fork's deliberate changes. Constraints any port must respect: branches not worktrees, controller-commits-only (via `commit-message` skill), resume-same-implementer for review fixes (upstream independently adopted the same rule in v6.0.0 — the divergence is validated, not a drift to fix), and the fork's own skills (external-review, using-stitch, …).
